Anatomy of a workflow file

push / pull_request triggers
The two most common triggers. Control exactly which branches, tags, and file paths activate the workflow.

push

Fires when commits are pushed to the repo. Use branches, tags, and paths to narrow it down.
on: push: branches: # only these branches - main - api - 'feature/**' # glob — any branch starting with feature/ - 'release/v[0-9]+' # regex-style glob branches-ignore: # skip these branches (cannot use with branches:) - 'dependabot/**' tags: # trigger on tag push - 'v*' # v1.0.0, v2.1.3 etc. paths: # only when these files change - 'backend/**' - 'requirements.txt' paths-ignore: # skip if only these files change - '**.md' - '.gitignore'

pull_request

Fires on PR events. Default activity types are opened, synchronize, reopened. Add types: to control which PR actions trigger the workflow.
on: pull_request: branches: [main] # only PRs targeting main types: # which PR events fire this - opened # PR first created - synchronize # new commits pushed to PR branch - reopened # closed PR reopened - closed # PR closed (merged or not) - labeled # label added to PR - ready_for_review # draft → ready - review_requested # reviewer added
⚠ pull_request vs pull_request_target — use pull_request for CI tests (no secret access for forked PRs). Use pull_request_target only if you need secrets in a fork PR context — it runs in the base repo context.

schedule — cron triggers
Run a workflow automatically on a time schedule using cron syntax. Useful for nightly tests, cleanup jobs, or health checks.
on: schedule: - cron: '0 2 * * *' # every day at 2:00 AM UTC - cron: '0 9 * * 1-5' # weekdays at 9 AM UTC - cron: '*/15 * * * *' # every 15 minutes - cron: '0 0 1 * *' # first day of every month
Cron format: minute hour day-of-month month day-of-week. All times are UTC. Minimum interval is every 5 minutes. Scheduled workflows only run on the default branch.

0 0 * * 0Every Sunday at midnight

0 6,18 * * *Twice a day — 06:00 and 18:00
workflow_dispatch — manual trigger
Adds a "Run workflow" button to the GitHub Actions UI. You can define inputs so the person running it can pass parameters.
on: workflow_dispatch: inputs: environment: description: 'Deploy target' required: true default: 'staging' type: choice options: [staging, production] run_tests: description: 'Run tests before deploy?' required: false type: boolean default: true version_tag: description: 'Version to deploy (e.g. v1.2.3)' type: string # Access inputs inside the workflow: jobs: deploy: runs-on: ubuntu-latest steps: - run: echo "Deploying ${{ inputs.environment }}" - if: inputs.run_tests == true run: pytest

workflow_run — trigger after another workflow

on: workflow_run: workflows: ["CI Pipeline"] # name of the other workflow types: [completed] branches: [main] jobs: deploy: if: github.event.workflow_run.conclusion == 'success' runs-on: ubuntu-latest
This is how you properly split CI and CD into separate workflow files — CD workflow_run fires when the CI workflow finishes successfully.

Self-hosted runners

runs-on: [self-hosted, linux, x64] # your own machine runs-on: [self-hosted, gpu] # custom label you gave the runner
Use self-hosted runners for private network access, GPU jobs, Windows with specific software, or cost savings on heavy workloads.

Matrix — run same job on multiple OSes

jobs: test: strategy: matrix: os: [ubuntu-latest, windows-latest, macos-latest] runs-on: ${{ matrix.os }}
needs: — job dependencies
Controls the order jobs run. Without needs:, all jobs run in parallel. With needs:, a job waits for others to finish first.
jobs: lint: # runs immediately runs-on: ubuntu-latest unit-tests: # runs immediately (parallel with lint) runs-on: ubuntu-latest integration-tests: needs: unit-tests # waits for unit-tests runs-on: ubuntu-latest docker-build: needs: [lint, integration-tests] # waits for BOTH runs-on: ubuntu-latest deploy: needs: docker-build if: always() # run even if docker-build failed
By default a job with needs: only runs if all its dependencies succeeded. Use if: always() to run regardless, or if: failure() to run only on failure.

Passing data between jobs with outputs

jobs: build: runs-on: ubuntu-latest outputs: image_tag: ${{ steps.meta.outputs.tags }} steps: - id: meta run: echo "tags=myimage:latest" >> $GITHUB_OUTPUT deploy: needs: build runs-on: ubuntu-latest steps: - run: echo "Deploy ${{ needs.build.outputs.image_tag }}"
if: — conditions
Control whether a job or step runs based on branch, event type, previous job result, or any context value.

Common if: patterns

# Branch conditions if: github.ref == 'refs/heads/main' if: github.ref != 'refs/heads/main' if: startsWith(github.ref, 'refs/heads/feature/') if: startsWith(github.ref, 'refs/tags/v') # Event type conditions if: github.event_name == 'push' if: github.event_name == 'pull_request' if: github.event_name == 'workflow_dispatch' if: github.event_name != 'pull_request' # PR merge check if: github.event.pull_request.merged == true # Previous job result if: success() # default — runs if all needed jobs passed if: failure() # runs only if a previous step/job failed if: always() # always runs, even after failure if: cancelled() # runs only if workflow was cancelled # Specific job result if: needs.unit-tests.result == 'success' if: needs.unit-tests.result == 'failure' if: needs.unit-tests.result == 'skipped' # Actor / environment if: github.actor == 'dependabot[bot]' if: github.actor != 'dependabot[bot]' # Manual input (workflow_dispatch) if: inputs.run_tests == true if: inputs.environment == 'production' # Combine conditions if: github.ref == 'refs/heads/main' && github.event_name == 'push' if: github.event_name == 'push' || github.event_name == 'workflow_dispatch' if: !startsWith(github.ref, 'refs/heads/dependabot')
FunctionReturns true when
success()All previous steps/jobs succeeded (default)
failure()At least one previous step/job failed
always()Always — regardless of any outcome
cancelled()Workflow was manually cancelled
startsWith(str, prefix)String starts with prefix
endsWith(str, suffix)String ends with suffix
contains(str, substring)String contains substring
matrix strategy — parallel job variants
Run the same job multiple times with different values. GitHub runs all combinations in parallel.
jobs: test: strategy: fail-fast: false # don't cancel others if one fails max-parallel: 3 # limit concurrent jobs matrix: python-version: ['3.11', '3.12'] os: [ubuntu-latest, windows-latest] exclude: # skip specific combos - os: windows-latest python-version: '3.11' include: # add extra combos - os: ubuntu-latest python-version: '3.13' experimental: true runs-on: ${{ matrix.os }} steps: - uses: actions/setup-python@v5 with: python-version: ${{ matrix.python-version }}
environment: — deployment targets
Named environments link a job to a configured GitHub Environment — giving access to environment-specific secrets, required reviewers, and deployment history.
jobs: deploy-staging: runs-on: ubuntu-latest environment: name: staging url: https://staging.yourapp.com # shown as link in Actions UI deploy-production: needs: deploy-staging runs-on: ubuntu-latest environment: name: production # triggers approval gate if configured url: https://yourapp.com steps: - run: echo ${{ secrets.PROD_DB_URL }} # env-specific secret
Set up approval gates at Settings → Environments → production → Required reviewers. The workflow pauses and emails the reviewer before the job runs.
permissions: — token access control
Controls what the auto-provided GITHUB_TOKEN can do. Default is read-only. Add write permissions only for jobs that need them.
# Workflow-level (applies to all jobs) permissions: contents: read # Job-level (overrides workflow-level for this job) jobs: create-pr: permissions: contents: write # push commits pull-requests: write # create / update PRs packages: write # push to GHCR issues: write # create / update issues statuses: write # set commit status checks: write # create check runs deployments: write # create deployments id-token: write # OIDC token for AWS/GCP keyless auth
PermissionNeeded for
contents: writePush commits, create releases, upload assets
pull-requests: writeCreate/update/merge PRs, add labels
packages: writePush Docker images to GHCR
issues: writeCreate/update/close issues
id-token: writeKeyless AWS/GCP auth via OIDC (no stored secrets)
deployments: writeCreate deployment records in GitHub
checks: writeCreate custom check runs and annotations
uses: — reusable actions
Pull in pre-built actions instead of writing shell commands. Four source formats.
# Format 1: Public GitHub action (owner/repo@ref) uses: actions/checkout@v4 uses: docker/build-push-action@v6 # Format 2: Specific commit SHA (most secure — pin to exact code) uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683 # Format 3: Local action in same repo uses: ./.github/actions/my-custom-action # Format 4: Reusable workflow in same repo uses: ./.github/workflows/shared-tests.yml # Format 5: Docker container as action uses: docker://alpine:3.19

run: — shell commands
Execute shell commands directly on the runner. Default shell is bash on Linux/Mac, PowerShell on Windows.
steps: # Single line - run: npm test # Multi-line with pipe - run: | echo "Installing..." pip install -r requirements.txt pytest --tb=short # Choose shell explicitly - run: Get-Date shell: pwsh # PowerShell - run: echo "bash only" shell: bash # Set working directory - run: pytest working-directory: ./backend # Write to GITHUB_OUTPUT (pass values to next steps) - id: version run: echo "tag=v1.0.0" >> $GITHUB_OUTPUT # Write to GITHUB_ENV (set env var for all following steps) - run: echo "NODE_ENV=production" >> $GITHUB_ENV # Continue even if this step fails - run: flake8 . || true continue-on-error: true

with: and env:
with: passes inputs to actions. env: sets environment variables for steps or jobs.
steps: # with: passes named inputs to a `uses:` action - uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' - uses: docker/build-push-action@v6 with: context: . push: true platforms: linux/amd64,linux/arm64 tags: myimage:latest # env: on a step — only available in that step - run: python deploy.py env: DATABASE_URL: ${{ secrets.DATABASE_URL }} APP_ENV: production # env: on a job — available to all steps in the job jobs: test: runs-on: ubuntu-latest env: PYTHON_ENV: test API_URL: http://localhost:8000 # env: at workflow level — available to all jobs env: REGISTRY: ghcr.io IMAGE_NAME: ${{ github.repository }}
outputs / inputs — sharing data
Pass values between steps within a job, and between jobs using outputs.
jobs: build: runs-on: ubuntu-latest outputs: # declare job outputs version: ${{ steps.get_ver.outputs.version }} image: ${{ steps.meta.outputs.tags }} steps: - id: get_ver run: | VER=$(cat version.txt) echo "version=$VER" >> $GITHUB_OUTPUT # Use in a later step of the SAME job - run: echo "Version is ${{ steps.get_ver.outputs.version }}" deploy: needs: build runs-on: ubuntu-latest steps: # Use output from ANOTHER job via needs context - run: echo "Deploy ${{ needs.build.outputs.image }}"

Secrets & variables
Encrypted secrets for sensitive values. Plain variables for non-sensitive config.
# Secrets — encrypted, masked in logs - run: deploy.sh env: DB_URL: ${{ secrets.DATABASE_URL }} SSH_KEY: ${{ secrets.SERVER_SSH_KEY }} TOKEN: ${{ secrets.GITHUB_TOKEN }} # auto-provided # Variables — plain text, visible in logs - run: echo ${{ vars.APP_ENV }} - run: echo ${{ vars.NODE_VERSION }} # Environment-specific secret (only available when # the job declares environment: production) - run: echo ${{ secrets.PROD_API_KEY }}
🔒 Never print a secret with echo $SECRET — GitHub will mask it in logs but it's still bad practice. Pass secrets via env: instead of shell variables.

Full workflow example
A complete CI/CD workflow using everything covered in this reference.
name: Full CI/CD Pipeline on: push: branches: [api, main] paths-ignore: ['**.md'] pull_request: branches: [main] workflow_dispatch: inputs: deploy_env: type: choice options: [staging, production] default: staging env: REGISTRY: ghcr.io IMAGE: ${{ github.repository }} jobs: ci: name: Run tests runs-on: ubuntu-latest strategy: matrix: python: ['3.11', '3.12'] steps: - uses: actions/checkout@v4 - uses: actions/setup-python@v5 with: {python-version: "${{ matrix.python }}", cache: pip} - run: pip install -r backend/requirements.txt - run: pytest backend/test -v --tb=short - uses: actions/upload-artifact@v4 if: always() with: {name: report, path: report.xml} auto-pr: needs: ci runs-on: ubuntu-latest if: github.event_name == 'push' && github.ref == 'refs/heads/api' permissions: {contents: write, pull-requests: write} steps: - uses: actions/checkout@v4 - uses: peter-evans/create-pull-request@v6 with: token: ${{ secrets.GITHUB_TOKEN }} base: main title: "✅ CI passed — ${{ github.sha }}" docker: needs: ci runs-on: ubuntu-latest if: github.ref == 'refs/heads/main' && github.event_name == 'push' permissions: {contents: read, packages: write} steps: - uses: actions/checkout@v4 - uses: docker/setup-qemu-action@v3 - uses: docker/setup-buildx-action@v3 - uses: docker/login-action@v3 with: registry: ghcr.io username: ${{ github.actor }} password: ${{ secrets.GITHUB_TOKEN }} - uses: docker/build-push-action@v6 with: push: true platforms: linux/amd64,linux/arm64 tags: ghcr.io/${{ github.repository }}:main cache-from: type=gha cache-to: type=gha,mode=max deploy: needs: docker runs-on: ubuntu-latest environment: name: production url: https://yourapp.com steps: - uses: appleboy/ssh-action@v1 with: host: ${{ secrets.SERVER_HOST }} username: ${{ secrets.SERVER_USER }} key: ${{ secrets.SERVER_SSH_KEY }} script: | docker pull ghcr.io/${{ github.repository }}:main docker stop app || true docker run -d --name app --restart unless-stopped \ -p 8000:8000 ghcr.io/${{ github.repository }}:main curl -f http://localhost:8000/health
